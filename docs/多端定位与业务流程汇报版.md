# 天府通未来教育中心 · 多端定位与业务流程汇报版

## 1. 项目一句话

天府通未来教育中心是面向校内课后延时服务的综合运营平台：平台把审核通过的机构课程引入合作学校，家长在天府通入口报名缴费，资金进入平台监管账户；课程按节上课、按节确认销课，月底按销课结果与机构结算。

核心价值是三件事：

- 对家长：只看到本校可报课程，报名缴费有平台托管，课程过程可追踪。
- 对学校：有课程资源但保留校内场地、排课、上课事实确认权。
- 对平台：掌握审核、分发、资金监管、销课确认、售后和结算的全链路。

## 2. 五个端的定位

| 端 | 使用对象 | 核心定位 | 负责什么 | 不负责什么 |
|---|---|---|---|---|
| 家长端 | 学生家长 | 报名与服务体验入口 | 看本校课程、选班报名、支付托管、看上课安排、看学习成果、确认结课、发起售后 | 不审核课程、不选机构后台流程、不参与结算 |
| 平台管理后台 | 平台运营、审核、财务、客服 | 全链路总控台 | 学校/场地底盘、机构审核、老师审核、课程审核、课程分发、订单监管、销课复核、售后仲裁、结算打款 | 不直接上课、不维护机构内部教学执行 |
| 学校端 | 学校管理员、教务人员 | 本校课后服务协同台 | 本校场地维护、确认课程开放、查看报名名单、确认排课、查看上课记录、确认上课事实、协同售后 | 不审核机构资质、不审核课程内容、不看机构结算金额 |
| 机构端 | 机构管理员 | 机构业务管理台 | 维护机构资料、创建老师、发布课程、查看分发班级和学生、跟踪销课、查看结算单 | 不直接把课程上架给家长、不接触平台托管资金 |
| 老师端 | 机构老师 | 一线上课执行台 | 查看自己班级、上课签到、学生考勤、提交课堂记录、提交结课评价 | 不看机构财务、不管理其它老师/课程、不处理结算 |

## 3. 各端的业务边界

### 家长端：前台交易和服务体验

家长从天府通进入未来教育服务，系统识别孩子所在学校，只展示这个学校已经开放的课程。家长报名的是具体班级，而不是抽象课程；班级包含时间、场地、老师、名额和价格。

家长完成报名支付后，资金进入平台监管账户。后续家长可以查看报名状态、上课安排、课程成果和售后进度。结课后可查看老师评价并确认课程完成。

### 平台管理后台：规则、审核、分发、资金的总控

平台后台是最高权限端，负责把业务跑通。它先维护学校与场地，再审核机构、老师和课程。课程审核通过后进入课程库，但课程库里的课程不会自动出现在家长端，必须由平台分发到指定学校。

平台还负责订单监管、销课异常处理、售后仲裁和结算打款。平台的核心权力是“谁能进来、课程给哪个学校、哪节课能计费、钱怎么结”。

### 学校端：本校资源和事实确认

学校端是这次新增后非常关键的一环。它不管机构资质和钱，但管本校是否开放课程、教室是否可用、排课是否冲突、老师是否真的来了、课是否真的上了。

学校确认课程开放后，该课程才在本校家长端可见；班级达到成班人数后，学校确认教室和时间；每节课结束后，学校可确认上课事实或标记异常。这个确认会影响机构是否能销课和结算。

### 机构端：课程供给和经营结果

机构端负责提供课程和老师。机构创建老师后要提交平台审核，通过后才能带班；机构发布课程也要经过平台审核，通过后进入课程库。机构不能直接对家长上架课程，课程是否进入某个学校由平台分发决定。

机构可查看被分发后的班级、学生、上课记录、销课状态和结算单。它关注的是“课程有没有被分发、班级有没有开、课有没有被确认、钱什么时候结”。

### 老师端：教学执行和销课源头

老师端是机构端下的子角色，只看自己负责的班级。老师到校后完成签到，给学生做考勤，上完课后提交课堂记录。课堂记录是后续生成销课记录的重要依据。

老师不看金额，也不处理机构结算。这样可以做到权限隔离：老师只负责教学过程，机构管理员负责经营管理，平台负责监管结算。

## 4. 业务主流程

### 阶段一：搭建资源底盘

```text
平台创建学校档案
→ 学校维护本校场地、容量、可用时间
→ 机构申请入驻
→ 平台审核机构资质
→ 机构创建老师
→ 平台审核老师资质
→ 机构发布课程
→ 平台审核课程内容
→ 课程进入平台课程库
```

### 阶段二：课程分发与招生

```text
平台从课程库选择课程
→ 分发到指定学校
→ 配置班级、时间、场地、价格、名额
→ 学校确认本校是否开放
→ 家长端展示该校可报课程
→ 家长选择具体班级报名
→ 家长完成支付，资金进入平台监管账户
→ 达到最低成班人数后成班
→ 学校确认排课和教室
```

### 阶段三：上课与销课

```text
老师到校签到
→ 学生考勤
→ 老师提交课堂记录
→ 系统生成销课记录：已上课待确认
→ 学校/平台确认上课事实
→ 变为已确认销课
→ 本节课进入机构可结算范围
```

如果学校发现老师迟到、未签到、实到人数异常、课程未正常开展，可标记异常。异常节次暂停计费，由平台介入，机构补充说明后再复核。

### 阶段四：结课、售后与结算

```text
班级完成全部课时
→ 老师提交结课评价/学习成果
→ 家长查看并确认
→ 如有问题，家长发起售后
→ 平台协调机构和学校处理
→ 月底按已确认销课生成机构结算单
→ 财务审核
→ 平台打款给机构
```

结算公式：

```text
机构实收 = 已确认销课金额合计 - 平台服务费 - 学校分成 - 退款扣减
```

|---|---|---|---|---|---|
| 学校与场地 | 查看所在学校 | 创建学校、监管配置 | 维护本校场地 | 查看分发结果 | 查看上课地点 |
| 机构入驻 | 不参与 | 审核机构 | 不参与 | 提交资料 | 不参与 |
| 老师资质 | 不参与 | 审核老师 | 查看任课老师 | 创建老师并提交审核 | 提供资质 |

| 课程分发 | 只看到本校课程 | 分发课程到学校 | 确认开放 | 查看分发学校 | 查看所带班级 |
| 报名支付 | 报名、支付、查看状态 | 监管订单/退款 | 查看本校名单 | 查看学生名单 | 查看自己学生 |
| 排课成班 | 查看上课安排 | 配置与监管 | 确认教室/时间 | 查看班级 | 按排课上课 |

## 6. 汇报时建议强调的三个闭环

### 第一，准入闭环

机构、老师、课程都要先审后用。机构审核通过才能开展业务，老师审核通过才能授课，课程审核通过才能进入课程库。

### 第二，校内服务闭环

课程不是平台审核完就直接给家长看，而是要分发到具体学校，并由学校确认开放。班级成班后，学校还要确认排课与场地，保证课后延时服务发生在可控的校内场景。

### 第三，资金与销课闭环

家长先支付，资金进入平台监管账户；老师每上一节课提交记录，学校/平台确认后才形成有效销课；月底只按已确认销课和扣减项给机构结算。

## 7. 一段汇报话术

这个平台不是单纯的报名小程序，而是课后延时服务的全链路管理系统。家长端解决选课报名和服务体验，机构端解决课程与老师供给，老师端负责上课执行，学校端负责校内资源和上课事实确认，平台后台负责审核、分发、监管、售后和结算。

整个业务的核心是“平台审、学校确认、家长支付托管、老师上课留痕、按节销课、按月结算”。这样既能保证学校场景可控，也能保证家长资金安全，还能让机构按真实完成的课时获得收入。
